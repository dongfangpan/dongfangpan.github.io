SDS vs C字符串
优点

  1. 常数复杂度获取字符串长度 O(1)，因为 len 属性中记录了 SDS 本身的长度。C 字符串是 O(N)
  2. 杜绝缓冲区溢出。
  3. 减少修改字符串长度时所需的内存重分配次数，SDS 修改字符串长度为 N 次最多需要执行 N 次内存重分配。C 字符串是必定执行 N 次内存重分配
  4. 二进制安全, SDS 可以保存文本或二进制数据。
  5. 兼容部分 C 字符串函数<string.h>。

链表

  • 每个链表节点都有一个指向前置节点和后置节点的指针，所以Redis的链表实现是双端链表
  • 链表表头节点的前置节点和表尾节点的后置节点都指向null，所以Redis是无环链表
  • 带表头指针和表尾指针：获取表头表尾数据时间复杂度为 O(1)
  • 带链表长度的计数器：获取长度时间复杂度 O(1)
  • 多态：链表可以保存不同类型的值
    head: 表头指针
    tail: 表尾指针
    len: 链表长度计数器
    dup、free、match: 函数

字典

  • 字典是一种用于保存键值对（key-value pair）的抽象数据结构。 table: 哈希表数组
    size: 哈希表大小
    sizemask: 哈希表大小掩码，用于计算索引值
    used: 该哈希表已使用节点的数量 哈希值的计算 Redis的哈希表使用链地址法来解决键冲突。

Rehash

  1. 扩容：ht[0] 为原始redis空间，ht[1] 为扩容后的 redis 空间，ht[1]的大小为第一个大于等于 ht[0].used*2。（ht[0].used*2 必须等于2n）
  2. 将保存在 ht[0] 上的所有键值对 rehash 到 ht[1] 上，重新计算键的哈希值和索引值
  3. 释放 ht[0]，把 ht[1] 设置为 ht[0]，并为 ht[1] 新创建一个空白哈希表为下次 rehash 作准备

rehash情况

  1. 无 BGSAVE 或 BGREWRITEAOF 操作时：哈希表负载因子>=1
  2. 有 BGSAVE 或 BGREWRITEAOF 操作时：哈希表负载因子>=5
  3. 哈希表负载因子<0.1，程序自动对哈希表执行收缩操作

负载因子 = 哈希表已保存节点数量/哈希表大小

渐进式rehash

  1. 为ht[1]分配空间，让字典同时持有ht[0]和ht[1]
  2. 设置一个计数器rehashidx=0，表示rehash开始
  3. 在rehash期间，字典执行添加、删除、查找或更新时会同时对两张表进行操作，rehashidx++
  4. 当ht[0]所有键值对rehash至ht[1]中时，rehashidx=-1，代表rehash操作完成

在渐进式rehash进行期间，字典的删除、查找、更新操作会在两个哈希表上进行，如查找，会先在ht[0]里查找，找不到再去ht[1]里查找；添加的操作只会在ht[1]里进行。

跳跃表

跳跃表支持平均O(logN)、最坏O(N)复杂度的节点查找。

head: 指向跳跃表的表头节点
tail: 指向跳跃表的表尾节点
level: 层数最大的那个节点的层数（表头节点不计）
length: 跳跃表目前包含节点的数量（表头节点不计）

L1, L2, L3: 层有2个属性：前进指针和跨度
BW: backward后退指针
1.0, 2.0, 3.0: 节点所保存的分值（score）
o1, o2, o3: 成员对象（obj）

整数集合

整数集合是集合键的底层实现之一，底层实现是数组，这个数组以有序、无重复的方式保存集合元素，再有需要时，程序会根据新添加元素的类型改变这个数组的类型。这个操作称之为升级，尽可能地节约了内存。

升级

  1. 根据新元素的类型，扩展整数集合底层数组的空间大小，并为新元素分配空间
  2. 将底层数组类型转换与新元素相同的类型，并将这些元素放到正确的位置，保证有序性质不变
  3. 将新元素添加到底层数组里
  • 整数集合不支持降级操作

内存回收与对象共享

内存回收

  • 在创建一个新对象时，引用计数的值会被初始化为 1；
  • 当对象被一个新程序使用时，它的引用计数值会被增一；
  • 当对象不再被一个程序使用时，它的引用计数值会被减一；
  • 当对象的引用计数值变为 0 时，对象所占用的内存会被释放。

对象共享

  1. 将数据库键的值指针指向一个现有的值对象
  2. 将被共享的值对象的引用计数增加 1

共享对象除了可以在字符串键使用，还可以在嵌套了字符串对象的对象（linkedlist 列表对象、hashtable哈希对象、hashtable集合对象、zset有序集合对象）使用。

算法导论-动态规划

Posted on 2019-10-21

分治策略（divide and conquer）

  1. 分解（divide）步骤将问题划分为一些子问题，子问题的形式与原问题一样，只是规模更小。
  2. 解决（conquer）步骤递归地解出子问题。如果子问题的规模足够小，则停止递归，直接求解。
  3. 合并（combine）步骤将子问题的解组合成原问题的解。
  • 递归情况：当子问题足够大，需要递归求解时
  • 基本情况：当子问题变得足够小，不再需要递归，我们说递归已经“触底”。

例子（merge-sort）

动态规划（dynamic programming）

动态规划（DP）通常用来求解最优化问题（optimization problem）
这种问题可以有很多可行的解，每个解都有一个值，希望找到最优值（最大或最小）的解。称这样的解为问题的一个最优解（an optimal solution）,而不是最优解（the optimal solution）,因为可能有多个解都达到最优。

DP VS D&C

动态规划和分治相似，都是通过组合子问题的解来求解原问题。

  • programming指的是一种表格法（按照一定的步骤来解决问题），而非编写计算机程序。

D&C

  1. 把问题划分为互不相交的子问题
  2. 递归求解子问题
  3. 把解组合起来，求出原问题的解

DP
应用于子问题重叠的情况（子问题求解是递归进行的，将其划分为更小的子子问题
分治会做很多不需要的工作，反复求解公共子子问题；动态规划对每个子问题就求解一次，保存起来，避免了重复计算。

DP设计

  1. 刻画一个最优解的结构特征
  2. 递归定义最优解的值
  3. 计算最优解的值（自底向上或自顶向下）
  4. 构造一个最优解（如只需要最优解的值而非解本身，则可忽略）

效率差的原因
CUT-ROD反复地用相同的参数值对自身进行递归调用，即它反复求解相同的子问题

  • 这棵递归调用树共有2n2n个结点，其中有2n−12n−1个叶结点
    运行时间T(n)=2nT(n)=2n, 即CUT-ROD的运行时间为n的指数函数

思考&改进

  1. 朴素递归算法之所以效率低是因为它反复求解相同的子问题
  2. 动态规划按照求解顺序，对每个子问题只求解一次，并把结果保存下来
  3. 随后再次需要此子问题的解，只需要查找保存的结果，不必重复计算
  4. 用空间来换时间（time-memory-trade-off）

动态规划方法求解最优钢条切割问题
实现动态规划的两种等价方法：

  1. 带备忘的自顶向下法（top-down with memoization）
    在自然递归形式编写的过程中，保存每个子问题的解
    解决子问题时，先检查是否保存过此解
    如果已存，直接返回保存的值
    如果没存，常规方法结算子问题
  2. 自底向上法（bottom-up method）
    使得任何子问题的求解只依赖于更小的子问题的求解
    将子问题按规模排序，由小到大顺序进行求解
    求解某子问题，他所依赖的更小的子问题都已求解完毕
    每个子问题只需要求解一次，当我们求解它时，它的前提问题都已求解完成

自顶向下的处理方法：
在求解xx时需要直接递归调用自身来求解yy（子问题就是从xx到yy的一条边）–深度优先
自底向上的处理方法：
给定一个子问题xx，求解它之前先求解邻接它的子问题yy – 对于任何子问题，直至它依赖的所有子问题均已3求解完成，才会求解它。
