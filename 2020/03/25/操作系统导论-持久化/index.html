I/O

I/O设备

  • 寄存器：
    • 状态寄存器：可以读取并查看设备的当前状态
    • 命令寄存器：用于通知设备执行某个具体任务
    • 数据寄存器：将数据传给谁备货从设备接收数据
  • 协议的执行过程：
    1. 轮询设备：操作系统反复读取状态处理器，等待设备进入可以接受命令的状态。
    2. 操作系统下发数据到数据寄存器，如果主CPU参与磁盘的写入操作，这个操作就是I/O
    3. 操作系统将命令写入命令寄存器，设备执行命令。
    4. 操作系统不断轮询设备，等待并判断设备是否执行完成命令（成功或失败）。
  • 减少CPU开销
    • 中断：有了中断之后，CPU不再需要轮询设备，只需要向设备发出一个请求，然后就可以让对应进程睡眠，切换执行其他任务。当完成了操作，跑出了硬件中断，引发CPU跳转执行中断处理程序。
    • 例子：
    • 如果设备非常快，通常使用轮询，轮询次数少，可以省去中断的操作；如果设备很慢，则用中断；如果设备的速度未知或忽快忽慢，则可以以使用混合策略，先轮询几次，在进行中断。

磁盘驱动器

  • 磁盘的结构
  • 寻道
  • I/O时间图：首先寻道，然后等待转动延迟，最后传输。
  • I/O时间 = 寻道时间+旋转时间+传输时间
  • 磁盘调度策略
    • 最短寻道优先（SSTF）
    • 最短定位时间优先（SPTF）

RAID

  • RAID 0：磁盘1和磁盘2并行
    • 容量：提供了N个有用容量，顶级。
    • 可靠性：任何磁盘故障都会导致数据丢失。
    • 性能：非常好，并行使用所有磁盘。
  • RAID 1：镜像，使每块磁盘都多一个副本
    • 容量：减少了一半
    • 可靠性：一个磁盘坏了，可以用另一个，两个都坏了数据就丢失了。
    • 性能：多了复制的过程，需要完成2次物理写入，平均时间略高。
  • RAID 5：在每个磁盘中定义一个融于块，如果有一个磁盘坏了，可以通过其他磁盘算出这个磁盘的读写。
    • 容量：N-1
    • 可靠性：可以运行一个磁盘故障
    • 性能：取决于带宽
  • RAID 10：混合了RAID 1和RAID 0
    • 容量：减少一半
    • 可靠性：和RAID 1一样
    • 性能：和RAID 0基本一样

文件系统实现

  • VSFS 文件系统：
    • D：数据区域，数据块
    • I：inode，index node 索引节点
    • id：inode位图i & 数据位图d
    • S：超级块，superBlock，表示该特定文件系统的信息：卷的大小、有多少inode、指向空闲列表块的头部指针等。
  • inode：每个inode都有一个数字（inumber）隐式调用，给定一个inumber，能直接计算磁盘上相对应节点的位置。
  • 多级索引：有一个间接指针，不是直接指向用户数据块，而是指向包含更多指针的块，每个指针指向用户数据。这个数据结构可以支持更大的文件。
  • 缓存和缓冲
    • 没有缓存，每个打开文件都需要对目录层次结构中的每一个级别至少进行2次读，一次读inode，一次读数据。
    • 读有缓存的文件：第一次读可能会产生很多I/O流量，但是随后打开同一文件大部分流量会命中缓存，不需要考虑I/O。
    • 写有缓存的文件：写缓冲优点
      • 通过延迟写入，文件系统可以将一些更新编成一批，放到一个小I/O中（比如把文件更新了又更新）
      • 通过将一些写入缓存在内存，系统可以调度后续I/O，提高性能。
      • 一些写入可以通过拖延避免。（比如写了又删）
    • 现在大多数现代文件系统都把缓冲设为5～30s，但是可以使用fsync()绕过缓存直接I/O

日志

  • 数据日志
    1. 日志写入：将事务的内容（包括TxB-事务开始和更新内容）写入日志，等待这些写入完成。
    2. 日志提交：将事务提交块（包括TxE-事务结束）写入日志，等待写完成，事务被认为已提交。
    3. 加检查点：将更新内容写入其最终的磁盘位置。（让它与日志中即将进行的更新一致）
    4. 释放：一段时间后，通过更新日志超级块，在日志中标记该事务为空闲。
  • 元数据日志
    1. 数据写入：将数据写入最终位置，等待完成。
    2. 日志元数据写入：将开始块和元数据写入日志，等待写入完成。
    3. 日志提交：将事务提交块（包括TxE-事务结束）写入日志，等待写完成，事务被认为已提交。
    4. 加检查点：将更新内容写入其最终的磁盘位置。
    5. 释放：一段时间后，通过更新日志超级块，在日志中标记该事务为空闲。
  • copy-on-write:
    • 原理
      • fork创建出的子进程，与父进程共享内存空间。也就是说，如果子进程不对内存空间进行写入操作的话，内存空间中的数据并不会复制给子进程，这样创建子进程的速度就很快了！(不用复制，直接引用父进程的物理空间)。
      • 并且如果在fork函数返回之后，子进程第一时间exec一个新的可执行映像，那么也不会浪费时间和内存空间了。
    • 另一种表达方式
      • 在fork之后exec之前两个进程用的是相同的物理空间（内存区），子进程的代码段、数据段、堆栈都是指向父进程的物理空间，也就是说，两者的虚拟空间不同，但其对应的物理空间是同一个。
      • 当父子进程中有更改相应段的行为发生时，再为子进程相应的段分配物理空间。
    • 原理实现
      • fork()之后，kernel把父进程中所有的内存页的权限都设为read-only，然后子进程的地址空间指向父进程。当父子进程都只读内存时，相安无事。当其中某个进程写内存时，CPU硬件检测到内存页是read-only的，于是触发页异常中断（page-fault），陷入kernel的一个中断例程。中断例程中，kernel就会把触发的异常的页复制一份，于是父子进程各自持有独立的一份。
    • 好处
      • COW技术可减少分配和复制大量资源时带来的瞬间延时。
      • COW技术可减少不必要的资源分配。比如fork进程时，并不是所有的页面都需要复制，父进程的代码段和只读数据段都不被允许修改，所以无需复制。
    • 缺点
      • 如果在fork()之后，父子进程都还需要继续进行写操作，那么会产生大量的分页错误(页异常中断page-fault)，这样就得不偿失。
    • 总结
      • fork出的子进程共享父进程的物理空间，当父子进程有内存写入操作时，read-only内存页发生中断，将触发的异常的内存页复制一份(其余的页还是共享父进程的)。
      • fork出的子进程功能实现和父进程是一样的。如果有需要，我们会用exec()把当前进程映像替换成新的进程文件，完成自己想要实现的功能。
