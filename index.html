分治策略（divide and conquer）

  1. 分解（divide）步骤将问题划分为一些子问题，子问题的形式与原问题一样，只是规模更小。
  2. 解决（conquer）步骤递归地解出子问题。如果子问题的规模足够小，则停止递归，直接求解。
  3. 合并（combine）步骤将子问题的解组合成原问题的解。
  • 递归情况：当子问题足够大，需要递归求解时
  • 基本情况：当子问题变得足够小，不再需要递归，我们说递归已经“触底”。

动态规划（dynamic programming）

动态规划（DP）通常用来求解最优化问题（optimization problem）
这种问题可以有很多可行的解，每个解都有一个值，希望找到最优值（最大或最小）的解。称这样的解为问题的一个最优解（an optimal solution）,而不是最优解（the optimal solution）,因为可能有多个解都达到最优。

DP VS D&C

动态规划和分治相似，都是通过组合子问题的解来求解原问题。

  • programming指的是一种表格法（按照一定的步骤来解决问题），而非编写计算机程序。

D&C

  1. 把问题划分为互不相交的子问题
  2. 递归求解子问题
  3. 把解组合起来，求出原问题的解

DP
应用于子问题重叠的情况（子问题求解是递归进行的，将其划分为更小的子子问题
分治会做很多不需要的工作，反复求解公共子子问题；动态规划对每个子问题就求解一次，保存起来，避免了重复计算。

DP设计

  1. 刻画一个最优解的结构特征
  2. 递归定义最优解的值
  3. 计算最优解的值（自底向上或自顶向下）
  4. 构造一个最优解（如只需要最优解的值而非解本身，则可忽略）

思考&改进

  1. 朴素递归算法之所以效率低是因为它反复求解相同的子问题
  2. 动态规划按照求解顺序，对每个子问题只求解一次，并把结果保存下来
  3. 随后再次需要此子问题的解，只需要查找保存的结果，不必重复计算
  4. 用空间来换时间（time-memory-trade-off）

动态规划方法求解最优钢条切割问题
实现动态规划的两种等价方法：

  1. 带备忘的自顶向下法（top-down with memoization）
    在自然递归形式编写的过程中，保存每个子问题的解
    解决子问题时，先检查是否保存过此解
    如果已存，直接返回保存的值
    如果没存，常规方法结算子问题
  2. 自底向上法（bottom-up method）
    使得任何子问题的求解只依赖于更小的子问题的求解
    将子问题按规模排序，由小到大顺序进行求解
    求解某子问题，他所依赖的更小的子问题都已求解完毕
    每个子问题只需要求解一次，当我们求解它时，它的前提问题都已求解完成
  • 两种方法得到的算法具有相同的渐进运行时间

    路由控制表中记录着网络地址与下一步应该发送至路由器的地址。
    在发送 IP包时,首先要确定IP包首部中的目标地址,再从路由控制表中找到与该地址具有相同网络地址的记录,根据该记录将IP包转发给相应的下一个路由器。如果路由控制表中存在多条相同网络地址的记录，就选择一个最为吻合的网络地址。
  8. DNS
    DNS 可以维护一个用来表示组织内部组织名（域名）和 IP 地址之间对应关系的数据库。当用户输入域名，DNS 会自动检测那个注册了主机名和 IP 地址的数据库，并迅速定位对应的 IP 地址。
  9. UDP 概念
    • UDP 不提供复杂的控制机制，利用 IP 提供面向无连接的通信服务。
    • UDP 是将应用程序发来的数据在收到的那一刻，立即按照原样发送到网络上的一种机制。即使是出现网络拥堵的情况，UDP 也无法进行流量控制等避免网络拥塞行为。
    • 传输途中出现丢包，UDP 也不负责重发。
    • 当包的到达顺序出现乱序时也没有纠正的功能
  10. UDP 使用场景
    1. 包总量较少的通信（DNS、SNMP等）
    2. 视频、音频等多媒体通信（即时通信）
    3. 限定于 LAN 等特定网络中的应用通信
    4. 广播通信（广播、多播）
  11. TCP 的三次握手和四次挥手
    • ACK（Acknowledge character）确认字符： TCP协议规定，只有ACK=1时有效，也规定连接建立后所有发送的报文的ACK必须为1
    • SYN（SYNchronization）： 在连接建立时用来同步序号。当SYN=1而ACK=0时，表明这是一个连接请求报文。对方若同意建立连接，则应在响应报文中使SYN=1和ACK=1. 因此, SYN置1就表示这是一个连接请求或连接接受报文。
    • FIN（finish）即完成、终结的意思， 用来释放一个连接。当 FIN = 1 时，表明此报文段的发送方的数据已经发送完毕，并要求释放连接。
  12. TCP 三次握手分别指
    （1）第一次握手：创建连接请求
    客户端发送创建连接请求报文段，该报文段的头部中SYN=1，seq=x。请求发送后，客户端便进入SYN-SENT状态。
    （2）第二次握手：服务器收到SYN报文段
    服务端收到连接请求报文段后，如果同意连接，则会发送一个应答：SYN=1，ACK=1，seq=y，ack=x+1。 此时服务器进入SYN_RECV状态。
    （3）第三次握手：客户端收到服务器的SYN+ACK报文段
    当客户端收到创建连接同意的应答后，还要向服务端发送一个确认报文段，告诉服务端发来的连接同意应答已经成功收到。
    该报文段的头部为：ACK=1，ack=y+1。
    客户端发完这个报文段后便进入ESTABLISHED状态，服务端收到这个确认应答后也进入ESTABLISHED状态，此时连接的建立完成。
  13. TCP 四次挥手分别指
    TCP连接是双向的，因此在四次挥手中，前两次挥手用于断开一个方向的连接，后两次挥手用于断开另一方向的连接。
    （1）第一次挥手：客户端向服务端发送一个FIN报文段：FIN=1，seq=u，客户端进入FIN_WAIT_1状态，表示客户端没有数据要发送给服务端了。FIN=1，表示要关闭一个方向的连接
    （2）第二次挥手：服务端收到了客户端发送的FIN报文段，向客户端回一个ACK报文段，ACK=1，ack=u=1,服务端告诉客户端同意关闭请求，客户端进入FIN_WAIT_2状态。
    （3）第三次挥手：服务端向客户端发送FIN报文段，请求关闭连接，同时服务端进入LAST_ACK状态；
    （4）第四次挥手：客户端收到服务端发送的FIN报文段，向服务端发送ACK报文段，然后客户端进入TIME_WAIT状态；服务端收到客户端的ACK报文段以后，就关闭连接；此时，客户端等待2MSL后依然没有收到回复，则证明Server端已正常关闭，客户端也可以关闭连接了。
  14. 为什么连接建立需要三次握手，而不是两次握手？
    防止失效的连接请求报文段被服务端接收，从而产生错误。
    失效的连接请求：若客户端向服务端发送的连接请求丢失，客户端等待应答超时后就会再次发送连接请求，此时，上一个连接请求就是失效的。
    如果建立连接只需两次握手，客户端并没有太大的变化，仍然需要获得服务端的应答后才进入ESTABLISHED状态，而服务端在收到连接请求后就进入ESTABLISHED状态。
    如果网络拥塞，客户端发送的连接请求迟迟到不了服务端，客户端便超时重发请求，如果服务端正确接收并确认应答，双方便开始通信，通信结束后释放连接。
    如果那个失效的连接请求抵达了服务端，由于只有两次握手，服务端收到请求就会进入ESTABLISHED状态，等待发送数据或主动发送数据。但此时的客户端早已进入CLOSED状态，服务端将会一直等待下去，这样浪费服务端连接资源。
  15. 为什么连接的时候是三次握手，关闭的时候却是四次握手？
    创建连接时，当Server端收到Client端的SYN连接请求报文后，可以直接发送SYN+ACK报文。其中ACK报文是用来应答的，SYN报文是用来同步的。
    关闭连接时，当Server端收到FIN报文时，并不会立即关闭SOCKET，所以只能先回复一个ACK报文，告诉Client端，“FIN报文我收到了”。只有等到Server端所有的报文都发送完了，才能发送FIN报文，因此不能一起发送。故需要四步握手。
  16. 为什么客户端要先进入TIME-WAIT状态，等待2MSL（最大报文段生存时间)）时间后才进入CLOSED状态？
    为了保证服务端能收到客户端的确认应答，如果客户端发完确认应答后直接进入CLOSED状态，那么如果该应答丢失，服务端等待超时后就会重新发送连接释放请求，但此时客户端已经关闭了，不会作出任何响应，因此服务端永远无法正常关闭。
  17. TCP 和 UDP 的区别
    1. TCP是面向连接（如打电话要先拨号建立连接）;UDP是无连接的，即发送数据之前不需要建立连接
    2. TCP提供可靠的服务。也就是说，通过TCP连接传送的数据，无差错，不丢失，不重复，且按序到达;UDP尽最大努力交付，即不保证可靠交付
    3. TCP面向字节流，实际上是TCP把数据看成一连串无结构的字节流;UDP是面向报文的UDP没有拥塞控制，因此网络出现拥塞不会使源主机的发送速率降低（对实时应用很有用，如IP电话，实时视频会议等）
    4. 每一条TCP连接只能是点到点的;UDP支持一对一，一对多，多对一和多对多的交互通信
    5. TCP首部开销20字节;UDP的首部开销小，只有8个字节
    6. TCP的逻辑通信信道是全双工的可靠信道，UDP则是不可靠信道
  18. TCP 如何保证可靠性
    TCP 通过校验和、序列号（下一批应该发什么数据）、确认应答、重发机制（超时重发或三次确认重发）、连接管理（三次握手四次挥手）、滑动窗口等机制实现可靠行传输。
  19. 滑动窗口
    单个确认应答效率差，以一个更大的单位进行确认时，转发时间会被大幅度缩短。
    • 发送端主机在确认应答返回之前，必须把发送的数据给保存在缓冲区中，当数据发送后如期收到确认应答就可以不重发数据，这些数据就可以在缓存中清除。收到应答后，将滑动窗口滑倒确认应答中的序列号的位置，这样就能以顺序的多个段同时发送数据以提高效率。
  20. 重发机制
  21. 流控制
    它的具体操作是，接收端主机向发送端主机通知自己可以接收数据的大小，于是发送端会发送不超过这个限度的数据。该大小限度就被称作窗口大小。
  22. 拥塞控制
    在网络出现拥堵时，如果突然发送一个较大量的数据，极有可能会导致整个网络的瘫痪。TCP 为了防止该问题的出现，在通信一开始时就会通过一个叫做慢启动的算法得出的数值，对发送数据量进行控制。
    • 慢启动
      慢启动算法的基本思想是当TCP开始在一个网络中传输数据或发现数据丢失并开始重发时，首先慢慢的对网路实际容量进行试探，避免由于发送了过量的数据而导致阻塞。主机发送了一个报文后就要停下来等待应答，每收到一个应答，拥塞窗口就增加一段长度，直至等于设定的阈值。比如我们可以先让发送方发一个包，等这个包被 ack 之后，我们再发 2 个包，这 2 个被 ack 之后再发 4 个包，以此类推，让一次所发的包数量慢慢增加，这就是慢启动。
  23. 拥塞避免算法
    拥塞避免算法让拥塞窗口缓慢增长，即每经过一个往返时间RTT就把发送方的拥塞窗口cwnd加1，而不是加倍。这样拥塞窗口按线性规律缓慢增长。
  24. 拥塞控制和流量控制的区别
    • 拥塞控制：拥塞控制是作用于网络的，它是防止过多的数据注入到网络中，避免出现网络负载过大的情况；常用的方法就是：（1）慢开始、拥塞避免（2）快重传、快恢复。
    • 流量控制：流量控制是作用于接收者的，它是控制发送者的发送速度从而使接收者来得及接收，防止分组丢失的。

备份与恢复

备份方法

根据备份的方法不同可以分为：

  1. Hot Backup（热备份）
    数据库运行中直接备份，对正在运行的数据库操作没有任何影响。
  2. Cold Backup（冷备份）
    在数据库停止的情况下备份，冷备只需要复制相关的数据库物理文件（frm文件、共享空间文件、独立表空间文件–*.ibd、重做日志文件）即可。
  3. Warm Backup（温备份）
    也是在数据库运行中进行，但是对当前数据库会有所影响，如加全局锁以保证一致性。

根据备份后文件的内容可以分为：

  1. 逻辑备份
    内容一般为一条条 SQL，或者是表内实际数据。（如mysqldump、select * into outfile）可以观察到导出文件内容，适用于数据库升级以及迁移，但是恢复所需时间长。
  2. 裸文件备份
    指复制数据库的物理文件，可以在数据库运行或者停止时复制。（ibbackup、xtrabackup）恢复时间短。

根据备份数据库的内容可以分为：

  1. 完全备份
  2. 增量备份
  3. 日志备份

对 MySQL 数据库的二进制日志的备份，通过对一个完全备份进行二进制日志重做（replay）来完成恢复工作。

具体实现

热备

热备原理

  1. 记录备份开始时，InnoDB 存储引擎重做日志文件检查点的 LSN。
  2. 复制共享表空间文件以及独立表空间文件。（所有数据）
  3. 记录复制完空间文件后，InnoDB 存储引擎重做文件检查点的 LSN。
  4. 复制在备份时产生的重做日志。

复制（replication）

复制原理

  1. 主服务器 master 把数据更改记录到二进制文件日志 binlog 中。
  2. 从服务器 slave 把主服务器的二进制文件复制到自己的中继日志 relay log 中
  3. 从服务器重做中继日志中的日志，把更改应用到自己的数据库上，以达到数据库最终的一致性。

InnoDB事务

事务

事务种类：

  1. 扁平事务
    由 begin 开始，由 commit 或 rollback 结束，所有操作符合原子性，要么都执行，要么都回滚。
  2. 带有保存节点的扁平事务
    在事务中，隐式地设置了一个保存点，当问题出现时，保存点能用作内部的重启点，根据应用逻辑，决定是回到最近一个保存点还是其他更早的保存点。
  3. 链事务
    链事务在提交事务时，释放不需要的数据对象，将有必要的处理上下文隐式地传给下一个要开始的事务。链事务中的回滚也只限制与当前事务，即只能恢复到最近一个保存点。
  4. 嵌套事务
    一般不考，InnoDB 未采用。
  5. 分布式事务
    在分布式环境下运行的扁平事务，需要根据数据所在位置访问网络中的不同节点。同样需要 ACID 特性。

事务的实现

两个日志：

  1. redo log：重做日志，保证事务原子性（重做日志缓冲）和持久性（重做日志文件）。
  2. undo log：保证事务的一致性，用于回滚记录到某个特定的版本和MVCC。

redo log

  • 每次重做日志缓冲写入重做日志文件后，InnoDB 都需要调用一次 fsync 操作。由于 fsync 的效率取决于磁盘的性能，因此磁盘的性能决定了事务提交的性能，也就是数据库的性能。

  • 由

参数控制重做日志刷新到磁盘的策略

  • 1：默认值，表示事务提交时必须调用一次 fsync 操作
  • 0：表示提交事务时不进行写入重做日志文件操作，master thread 会每 1 秒进行一次重做日志文件的 fsync 操作。
  • 2：表示事务提交时将重做日志写入重做日志文件，但仅写入文件系统的缓存中，不进行 fsync 操作。

  • 为提高 fsync 效率，数据库提供了 group commit 功能，即一次 fsync 可以刷新确保多个事务日志被写入文件。操作：

    • 修改内存中事务对应的信息，并把日志写入重做日志缓冲。
    • 调用 fsync 将确保日志都从重做日志缓冲写入磁盘。

redo log 和 binlog 区别

redo log 重做日志 binlog 二进制日志
在 InnoDB 存储引擎层产生 在 MySQL 数据库的上层产生
物理格式日志，其记录的是对于每个页的修改 逻辑日志，其记录的是对应的 SQL 语句
在事务进行中不断地被写入 只是事务提交完成后进行一次写入
循环使用 不能循环使用，每次重启产生新的binlog文件
异常宕机或介质故障后的数据恢复使用 恢复数据时使用

LSN（log sequence number） 日志序列号作用：

  1. 记录重做日志写入总量。
  2. 确定 checkpoint 位置，对比数据库启动前后 LSN 变化。
  3. 页的版本。

恢复

  • InnoDB 存储引擎在启动时不管上次数据库运行时是否正常关闭，都会尝试进行恢复操作。redo log 记录的是物理日志，所以恢复速度会比 binlog 快很多。
  • 由于 checkpoint 表示已经刷新到磁盘上的页的 LSN，所以恢复过程中仅需恢复 checkpoint 开始的日志部分。

undo log

如果用户执行的事务或语句由于某种原因失败了，又或者用户用一条 rollback 语句请求回滚，就可以利用这些 undo 信息将数据回滚到修改之前的样子。

undo log 原理

  • 当 InnoDB 存储引擎回滚时，它实际上做的事与之前相反的工作。
  • undo 的另一个作用的是 MVCC，即 InnoDB 存储引擎中的 MVCC 的实现是通过 undo 实现的。
  • undo log 的产生会伴随着 redo log 的产生，因为 undo log 也需要持久性保护。

undo log 操作

  • 事务提交时，InnoDB存储引擎会做以下2件事：
    • 将 undo log 放入链表中，是否可以最终删除 undo log 和所在页由 purge 线程判断。
    • 判断 undo log 所在的页是否可以重用，若可以则分配给下一个事务使用。

undo log 类型

  • insert undo log：只对事物本身可见，其他事务看不见，所以 undo log 可以在事务提交后直接删除，不需要 purge 操作。
  • update undo log：是 delete 和 update 操作产生的 undo log；可能需要提供 MVCC 机制，因此不能在事务提交时就删除，提交时放入 undo log 链表，等待 purge 线程进行最后的删除。
    • delete操作实际上不会直接删除，而是将delete对象打上delete flag，标记为删除，最终的删除操作是purge线程完成的。
    • update分为两种情况：update的列是否是主键列:
      • 如果不是主键列，在undo log中直接反向记录是如何update的。即update是直接进行的。
      • 如果是主键列，update分两部执行：先删除该行，再插入一行目标行。

purge

purge 清理过程

  1. 找到队尾的 tx1
  2. 在 tx1 所在的 page1 中处理 tx1，tx3
  3. 处理到 tx5 的时候发现 tx5 在使用
  4. 重新去队列中找到队尾未处理的 tx2
  5. 处理 tx2 所在的 page2 中的 tx2，tx6，tx4

分布式事务（XA事务）

  • 使用分布式事务时，InnoDB 的事务隔离级别必须设置为 seriablizable。
  • XA 事务允许不同数据库之间的分布式事务，可以在 MySQL 数据库、Oracle 数据库或SQL server数据库之间创建，只需要参与在全局事务中的每个节点都支持 XA 事务。
  • 分布式事务的两段式提交
    1. 所有参与全局事务的节点都开始准备（prepare），告诉事务管理器它们准备好提交了。
    2. 事务管理器告诉资源管理器执行 rollbacks 还是 commit。
    3. 如果任何一个节点不能提交，则所有的节点都被告知需要回滚。

内部分布式事务

内部分布式事务存在于 binlog 和 InnoDB 存储引擎之间。在事务提交时，先写二进制日志binlog，再写重做日志redo log。这两个日志必须符合原子性，即需要同时写入。

过程：

  • 当事务提交时，InnoDB 存储引擎会先做一个 prepare 操作，将事务的 xid 写入，然后在进行二进制日志写入。
  • 若在 InnoDB 存储引擎提交前，MySQL 数据库宕机了，MySQL 数据库在重启后会先检查准备的 UXID 事务是否提交，若没有，则在存储引擎层在进行一次提交操作。

锁

InnoDB存储引擎提供一致性的非锁定读、行级锁支持。行级锁没有相关额外的开销，并可以同时得到并发性和一致性。且InnoDB不需要锁升级，因为一个锁和多个锁开销是一样的。

InnoDB锁

锁的类型

InnoDB实现了两种标准行级锁：

  1. 共享锁（S Lock），允许事务读一行数据。
  2. 排他锁（X Lock），允许事务修改或删除一行数据。

一致性非锁定读

一致性的非锁定读是指 InnoDB 存储引擎通过行多版本控制的方法来读取当前执行时间数据库中行的数据。

  • 如果读取的行正在执行 delete 或 update 操作，这时读取操作不会因此去等待行上锁的释放。相反，InnoDB 会去读取行的快照数据。
  • 快照数据是指该行的之前版本的数据，这是通过 undo 段来完成。undo 段是用来在事务中回滚数据，因此快照数据本身没有开销，也不用上锁。

READ COMMITTED 和 REPEATABLE READ 一致性非锁定读区别
READ COMMITTED：读取被锁定行的最新一份快照数据。
REPEATABLE READ：读取事务开始时的行数据版本。
例子：

时间 会话A 会话B
1 begin
2 select * from parent where id = 1;
3 begin
4 update parent set id=3 where id=1;
5 select * from parent where id = 1;
6 commit
7 select * from parent where id = 1;
8 commit

在第五步的时候，若使用 READ COMMITTED 会返回一个 empty set；若用 REPEATABLE READ 则还是返回id=1的行。

一致性锁定读

支持一致性锁定读的操作：

  1. select … for update：对读取的行记录加一个 X 锁，其他事务不能对已锁定的行加上任何锁。
  2. select … lock in share mode：对读取的行记录加一个 S 锁，其他事务可以对该行加 S 锁，但不能加 X 锁。

select … for update、select … lock in share mode 必须在一个事务中，当事务提交了，锁也就释放了。

锁的算法

行锁的3种算法：

  • Record Lock：单个行记录上的锁，锁住索引记录
  • Gap Lock：间隙锁，锁定一个范围，但不包含记录本身
  • Next-Key Lock：Gap+Record,锁定一个范围，并且锁定记录本身

Next-Key Lock 例子：
一个索引有 10、11、13、20 四个值，该索引就会被 Next-Key Lock 分为5个区间：(-∞,10]、(10,11]、(11,13]、(13,20]、(20,+∞]。

Next-Key Lock 降级算法
当索引含有唯一属性时，InnoDB 会对 Next-Key Lock 进行优化，将其降级为 Record Lock，仅锁住索引本身，而不是范围。

幻读

幻读是指在同一事务下，连续执行两次同样的SQL语句可能导致不同的结果，第二次的SQL语句可能会返回之前不存在的行
例子：

时间 会话A 会话B
1 set session tx_isolation=’read-committed’
2 begin
3 SELECT * FROM t WHERE a> 2 FOR UPDATE; // 1 row a:5
4 begin
5 insert into t select 4;
6 commit
7 SELECT * FROM t WHERE a> 2 FOR UPDATE; // 2 row a:4 a:5
8 commit

采用 Next-Key Lock 可以避免幻读，因为在 Next-Key Lock 模式下，SELECT * FROM t WHERE a> 2 FOR UPDATE 会对（2，+∞）加上一个 X 锁。

所以 Next-Key Lock 可以解决幻读的问题。

锁问题

  1. 脏读
    一个事务可以读到另外一个事务中未提交的数据。只有 read-uncommitted 模式下才会发生。
  2. 不可重复读（幻读）
    在一个事物内多次读取到同一个数据集合，但由于另一事务的修改，第一个事务两次读到的数据不一样。脏读是读为提交数据，幻读是读已经提交的数据。read-committed 模式下会发生幻读，read repeatable 可以避免幻读现象，因为采用了 Next-Key LocK。
  3. 丢失更新
    事务的更新操作会被另一个事务的更新操作所覆盖，从而导致数据不一致，避免的话需要让事务串行化，类似整个事务阶段都加锁。

死锁

死锁是指两个或多个事务在执行过程中，因争夺锁资源而造成的一种互相等待的现象。
解决死锁的办法

  1. 超时：当两个事务相互等待时，当一个等待时间超过阈值，则将这个事务回滚。
  2. 等待图：主动的死锁检测方式，InnoDB采用这种方式。这种机制会在每个事务请求锁并发生等待时都会判断是否有回路，若存在则有死锁，InnoDB 会选择回滚 undo 量最少的事务。如下图，t1 和 t2 就存在回路。（InnoDB1.2版本前使用dfs实现，之后用非递归方法实现）

B+ 树索引

B+ 树索引并不能找到一个给定键值的具体行，B+ 树索引能找到的只是被查找数据行所在的页。然后数据库通过把页读到内存，再在内存中进行查找，最后得到要查找的数据。

聚簇索引
在聚簇索引中寻找数据：

  1. 在非叶子节点中找出主键所在数据页，在数据页中找到主键位置
  2. 根据主键（Key）算出指向数据页的页号（Pointer）
  3. 根据 Pointer 在叶子节点上找到对应的页
  4. 在对应的页中找到主键对应的位置（页里面是按照主键顺序排列的双向链表）
  5. 返回主键对应的数据信息

InnoDB存储引擎

Posted on 2020-03-29

InnoDB存储引擎

InnoDB体系架构

后台线程

线程类型：

  1. Master Thread：负责将缓冲池中的数据异步刷新到磁盘，保证数据一致性（脏页刷新、合并插入缓冲、undo页回收）。
  2. IO Thread：InnoDB 用了大量 AIO（Async IO）处理写 IO 请求，IO Thread 负责这些 IO 请求的回调处理。
  3. Purge Thread：回收已经使用并分配的 undo 页。
  4. Page Cleaner Thread：减轻原 Master Thread 的工作，提高 InnoDB 存储性能。

内存

缓冲池

  • 缓冲区是一块内存区域，通过内存的速度来弥补磁盘速度较慢对数据库性能影响。
  • 数据库首次读取页时，会把从磁盘里读取的页放在缓冲池中，下次该页被缓冲池命中，则读取页，否则，在磁盘里中读取。
  • 对于数据库中的页的修改操作，先修改缓存池中的页，再以一定的频率刷到磁盘中。（Checkpoint 机制）
  • InnoDB内存数据对象

LRU list

  • 通常来说，数据库的缓冲池是通过 LRU 算法进行管理的（频繁使用的页在列表最前端，最少使用的页在尾端）。
  • 优化的 LRU 算法：新读到的页放到 LRU 列表的 midpoint 位置。（原因：当进行某些操作如索引或数据的扫描，可能会使缓冲池中的页被刷新出，影响缓冲池效率。这些操作访问的很多页都是仅在这次操作中需要，并不是活跃数据。）

LRU list vs Free list
数据库刚启动时，LRU 列表是空的，所有页都存在 Free 列表中。当需要从缓冲池中分页时，首先从 Free 列表中查找是否有空闲页，若有则将该页从 Free 列表中删除，放入到 LRU 列表中。

脏页和Flush list
在 LRU 列表中的页被修改后，称该页为脏页，即缓冲池中的页和磁盘上的页的数据产生了不一致。数据库会通过 CHECKPOINT 机制将脏页刷新回磁盘，Flush 列表中的页就是脏页列表。

重做日志缓冲、额外的内存池

重做日志缓冲：

  • InnoDB 存储引擎首先将重做日志信息放到这个缓冲区，然后按照一定的频率刷新到重做日志文件。
  • 一般情况下 重做日志缓冲池只需要 8M 就够了，因为重做日志会在下面三种情况把内存刷到外部磁盘的重做日志文件中：
    • Master Thread 每一秒就会进行刷新操作
    • 每个事务提交就会进行刷新操作
    • 重做日志缓冲池空间剩余1/2时会进行刷新操作。

额外的内存池：
对一些数据结构本身的内存进行分配时，需要从额外的内存池中进行申请，当该区域不够时，会向缓冲池中进行申请。

Checkpoint 技术

Checkpoint 技术的目的是为了解决：

  1. 缩短数据库的恢复时间
    当数据库发生宕机时，数据库不需要重做所有的日志，因为 Checkpoint 之前的页都已经刷新回磁盘。
  2. 缓冲池不够用时，将脏页刷新到磁盘
    当缓冲池不够用时，LRU 算法回溢出最近最少使用的页，如果这个页是脏页，就前置执行 Checkpoint，将脏页刷进磁盘。
  3. 重做日志不可用时，刷新脏页

Innodb 有2种 Checkpoint

  1. Sharp Checkpoint：发生在数据库关闭时将所有的脏页都刷新回磁盘。
  2. Fuzzy Checkpoint：进行页刷新时，只刷新一部分脏页。
    Master Thread Checkpoint、FLUSH_LRU_LIST Checkpoint、Async/Sync Flush Checkpoint、Dirty Page too much Checkpoint 都属于Fuzzy Checkpoint

InnoDB关键特性

插入缓冲

Insert Buffer

  • Insert Buffer 和数据页一样，都是物理页的一个组成部分，并不是缓冲池中的一部分。
  • 插入聚集索引（Primary Key）是顺序的不需要磁盘随机读；对于非聚集辅助索引（secondary index）叶子节点的插入不是顺序的，需要离散的访问非聚集索引页，导致插入操作新能下降。（有些辅助索引插入仍是顺序的，比如ctime）
  • 对于非聚集索引的插入或更新操作，不是每一次都直接插入到索引页中，而是先判断插入的非聚集索引页是否在缓冲池中，若在直接插入，若不在先放到 Insert Buffer 对象中。然后以一定的频率和情况进行 Insert Buffer 和辅助索引叶子节点的 merge 操作。
  • 使用 Insert Buffer 需要满足一下子2个条件
    1. 索引是辅助索引
    2. 索引不是唯一索引
  • 例子：2246304 次 merge 可以 merge 7540969数据，比例差不多是1:3，代表了插入缓冲将对于非聚聚索引页的离散 IO 逻辑请求大约降低了 2/3
  • Insert Buffer 的数据结构是一棵 B+ 树（非叶子节点存放的是查询的键值）

Change Buffer

  • Change Buffer 是 Insert Buffer 的升级，引入了 Delete Buffer 和 Purge Buffer。（对象依旧是非唯一的辅助索引）
  • Delete 操作可能分为2个过程：
    1. 将记录标记为已删除（Delete Buffer 操作）
    2. 真正将记录删除（Purge Buffer 操作）
  • SHOW ENGINE INNODB STATUS 展示：

    insert 表示 Insert Buffer；delete mark 表示 Delete Buffer；delete 表示 Purge Buffer

两次写

自适应哈希索引

  • InnoDB存储引擎会监控对表上各索引页的查询。如果观察到建立哈希索引可以带来速度提升,则建立哈希索引,称之为自适应哈希索引(Adaptive Hash Index,AHI) 。
  • 经常访问的二级索引数据（热点数据）会自动被生成到hash索引里面去(最近连续被访问三次的数据)，自适应哈希索引通过缓冲池的B+树构造而来，因此建立的速度很快。
  • AHI 要求：
    • 以该模式访问了100次
    • 页通过该模式访问了N次（N=页中记录 * 1/16）
  • AHI 缺点：
    1. hash自适应索引会占用innodb buffer pool；
    2. 自适应hash索引只适合搜索等值的查询（例如=、<=>、in），如select * from table where index_col=’xxx’，而对于其他查找类型，如范围查找，是不能使用的；
    3. 极端情况下，自适应hash索引才有比较大的意义，可以降低逻辑读。
    4. 无法用于排序
    5. 有冲突可能
    6. MySQL自动管理，人为无法干预。

异步 IO

采用异步 IO（Asynchronous IO，AIO）的方式来处理磁盘可以提高磁盘操作性能。

Sync IO vs AIO
Sync IO 每进行一次 IO 操作，需要等待此次操作结束才能继续接下来的操作；AIO 可以在发出一个 IO 请求后立即在发出另一个 IO 请求，当全部 IO 请求发送完毕后，等待所有 IO 操作的完成。

IO merge
将多个 IO 合并为一个 IO，以提高 IOPS 性能。
例子：用户需要访问页的（space, page_no）为：(8，6)、(8，7)、(8，8)。每页大小为16K，如果使用同步 IO 需要进行3次 IO 操作；而AIO 会判断这三个页是连续的，因此 AIO 底层会发送一个 IO 请求，从（8，6）开始读 48K 的页

刷新邻接页
当刷新一个脏页时，innodb存储引擎会检测该页所在区(extent)的所有页，如果是脏页，那么一起进行刷新。这样做的好处显而易见，通过AIO可以将多个IO写入操作合并为一个IO操作，增大写入量，减少了物理写IO，故该工作机制在传统机械磁盘下有着显著的优势。

事物的ACID性质

  1. 原子性：
    数据库将事务中的多个操作当作一个整体来执行，要么都执行，要么都不执行。Redis 不支持事物会滚，即使事务中某个命令出现了错误，整个事务也会继续执行，直到都执行完毕为止，执行完了错误也会一并打印出来。
  2. 一致性：
    数据库在执行之前是一致的（数据符合数据库本身的定义和要求，没有包含非法或无效的错误数据），那执行事务之后无论成功与否数据库也是一致的。
  3. 隔离性：
    多个事务并发执行，也不会互相影响。
  4. 持久性
    当事务执行完了之后，所得到的结果会保存到永久性储存介质（如硬盘）中，即使执行事务完毕之后停机了，结果也不会丢失。
    在 RDB 持久化模式下，需要执行 BGSAVE 才会对数据库进行保存，数据不能第一时间保存到硬盘里，所以 RDB 模式下的事务不具有持久性。
    在 AOF 持久化模式下，如果 appendfsync 设置为 always，则事务具有持久性，设置为 everysec 或者 no 时不具有持久性

  4. 主服务器向从服务器发送缓存区保存的写命令，从服务器执行这些命令
  5. 主服务器对从服务器执行命令传播操作

为什么要改进旧版主从复制过程中的断线就完整同步的操作：

新版主从复制改进
使用 PSYNC 替代 SYNC，具有完整重同步和部分重同步两种模式。
完整重同步用于首次复制情况，和旧版的一样。
部分重同步摒弃了旧版复制功能在断线重连之后需要重新全部同步的缺陷，在断线重连之后只需要同步之前没同步的那些命令就行了。

部分重同步功能的构成：

  1. 主从服务器的复制偏移量 - 主从服务器偏移量不一致则进行重同步。
  2. 主服务器的复制积压缓冲区 - 缓冲区是一个固定长度先进先出的队列，如果偏移量在缓冲区里面则进行部分重同步操作，否则进行完整重同步操作
  3. 运行ID - 从服务器保存的运行ID和当前连接的主服务器运行ID相同则进行部分重同步操作，否则进行完整重同步操作。

  1. 检测主服务器的网络连接状态，lag 值超过一秒证明主从服务器连接故障。

一个 Redis 集群通常由多个节点（node）组成，一开始每个节点都是独立的，各个独立的节点连接起来构成一个多节点集群。

节点启动
节点会继续使用所有在单机模式下使用的服务器组建，如时间处理器，事件处理器，RDB AOF 持久化模块等。

节点的握手过程

  1. 节点 A 为节点 B 创建一个 clusterNode 结构，并将该结构添加到自己的 clusterState.nodes 字典里
  2. 节点 A 发送 MEET 命令给节点 B
  3. 节点 B 接收到 MEET 命令，为节点 A 创建一个 clusterNode 结构，将该结构添加到自己的 clusterState.nodes 字典里
  4. 节点 B 向节点 A 发送 PONG 消息
  5. 节点 A 接收到 PONG 消息，向节点 B 发送 PING 消息

指派槽 slot

集群的整个数据库被分为 16384（214214）个槽，数据库中的每个键都属于着 16384 个槽的其中一个，集群中的每个节点可以处理 0～16384 个槽。
如果 16384 个槽都有节点在处理，集群处于上线状态（ok），否则，只要有一个槽没有节点处理，集群就处于下线状态（fail）。

故障检测

  1. 集群中每一个主节点都会定期向集群中的其他主节点发送 PING 消息，其他节点收到消息会返回一个 PONG 消息。
  2. 当一个节点发送 PING 之后的一定时间内未收到 PONG 消息，则把接收节点 x 标记为疑似下线节点。
  3. 当超过半数的主节点都把 x 节点标记为疑似下线节点，则 x 节点会被标记为下线。
  4. x 节点的其中一个从节点会竞选成为新的主节点（每个主节点都有投票权，当超过半数的主节点吧支持票投给其中一个从节点，那这个从节点就被选举为新的主节点）

消息

集群中的节点通过发送和接收消息来进行通信，常见的消息包括 MEET、PING、PONG、PUBLISH、FAIL 五种。
